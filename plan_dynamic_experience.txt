Plan: Dynamic Experience Flow Integration
========================================

Context Snapshot
----------------
Repository: interview-flow (AI-driven technical interview assistant)
Key modules in scope:
- gateway_service: routes named LLM tasks to providers with YAML-based overrides.
- orchestrator_service.llm_api: stage helpers (warm-up, evidence, theory, wrap-up) using DSPy and gateway tasks.
- session_service.interview_state: state machine tracking phases and step order.
- session_service.service: WebSocket manager that seeds Stage-0 context and loops through orchestrator decisions.
- orchestrator_service.programs.stage1_warmup / stage2_evidence: DSPy parsing programs that normalize candidate answers.
Current warm-up flow:
1. Stage-0 (`Stage0AnalysisProgram`) analyzes JD/resume and returns skill overlaps but does not choose specific projects.
2. `InterviewState` defines static warmup/evidence step lists. Orchestrator iterates them sequentially.
3. Warm-up helpers ask targeted questions to fill `ContextPacket.project_context` fields; evidence helpers follow to extract structured notes and hooks.
Goal:
- Allow Stage-0/LLM to pre-select the most relevant project/experience from the resume.
- Generate a dynamic plan of “experience slots” merging warm-up and evidence prompts so we can adapt order/coverage without breaking later stages.
- Maintain existing data contracts (`ContextPacket`, `project_context`, notes, hooks) so theory/wrap-up logic remains unchanged.

1. Stage-0 Enhancements
-----------------------
- Extend `JDResumeAnalysisOutput` and prompt to include:
  * `selected_project`: concise label or identifier for the resume project the LLM deems most relevant.
  * `experience_plan`: ordered list of slot identifiers (strings) representing questions to ask (e.g., `project_overview`, `architecture`, `components_list`).
- Implementation steps:
  * Update `stage0_analysis.py` prompt to request these keys and adjust the Pydantic model accordingly.
  * If Stage-0 cannot determine a confident project or plan, return `selected_project = None` and `experience_plan = []` so runtime can fall back to legacy behavior.
  * (Optional) Add a dedicated planner task (`warmup_evidence_plan`) invoked from Stage-0 helper if we want to separate concerns: Stage-0 collects raw signals, planner consumes them to emit plan. Keep gateway task definitions in `llm_router_config.yml` up to date.

2. Interview State & Orchestrator Refactor
-----------------------------------------
- Replace the fixed `warmup_steps`/`evidence_steps` arrays in `InterviewState` with:
  * `default_experience_steps`: concatenated legacy order for fallback use.
  * `experience_plan`: deque populated from Stage-0 output (or default if empty).
  * Helper methods `current_experience_step()` and `advance_experience_step()` to manage the queue, while exposing `current_phase` values like `experience`, `theory`, `wrap_up`.
- Update `ConnectionManager.handle_message` to inject Stage-0 plan into the state when initializing `InterviewState`.
- Modify `Orchestrator.decide_next_action` to:
  * Treat `experience` phase as the unified replacement for warm-up + evidence.
  * Dispatch based on the current slot identifier using a lookup that maps slot IDs to existing helper functions (e.g., `project_overview -> warmup_project_overview`, `components_list -> evidence_components_list`).
  * Preserve targeted follow-up injections before executing the next planned slot.
  * Continue logging `question_type` and stage metadata so UI/tests stay consistent. When running legacy fallback, produce the same sequence as today.

3. Helper & Test Adjustments
----------------------------
- Validate that helper functions already skip execution when context is populated (many do). Add explicit checks where missing so prefilled data from Stage-0 does not trigger redundant LLM calls.
- Ensure combined experience slots still decrement time and update `ContextPacket` exactly as before.
- Update unit tests:
  * Add coverage for planner-provided plan (mock Stage-0 to return `experience_plan`) ensuring orchestrator follows the custom order.
  * Keep existing tests (e.g., `test_end_to_end`, `test_followups`) passing by exercising fallback path or adjusting expected phase names if they change (`warm_up` -> `experience`).
  * Add assertions verifying that Stage-0-selected `selected_project` skips the initial project selection question.
- Documentation/todos:
  * Update `llm_router_config.yml` with new task(s) if necessary.
  * Note behavior changes in `plan_dynamic_experience.txt` and future dev docs.

- Confirm PDF/report generation still sees correct notes/hooks after merged flow.
